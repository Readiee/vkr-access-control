# 7. Интеграционное тестирование и стабилизация прототипа

## 7.1. Стратегия тестирования

Тестирование велось на трёх уровнях. Модульные тесты проверяют изолированные сервисы (политики, графовый анализ, кэш-менеджер, утилиты сериализации). Интеграционные тесты прогоняют сценарии через REST API: создание правила, проверка доступа, верификация курса, импорт структуры со связанным набором политик. Демонстрационные сценарии запускают полный конвейер логического вывода на подготовленных онтологиях с заранее известным ожидаемым результатом.

| Уровень | Файлов | Что проверяется |
|---|---|---|
| Модульные тесты | 6 | Политики, граф зависимостей, агрегация, кэш, сериализаторы |
| Интеграционные тесты | 7 | Контракты REST API, конвейер логического вывода через FastAPI TestClient |
| Демо-сценарии | 8 | Корректность вывода на happy_path и семи bad_sv-сценариях |

Общее число прогоняемых тестов составляет 117. Покрытие по строкам исходного кода — 84%, по ядру (сервисы логического вывода и верификации) — не ниже 82%.

## 7.2. Демонстрационные сценарии

Для прогона интеграционного контура подготовлены восемь сценариев. Базовый `happy_path` описывает корректный учебный курс: четыре модуля, четыре обучающихся, набор политик, покрывающий все девять типов. Семь сценариев `bad_sv*` содержат внедрённые дефекты, по одному на класс верифицируемого свойства, с эталонной разметкой ожидаемого результата.

| Сценарий | Класс дефекта | Ожидание |
|---|---|---|
| `happy_path` | — | Все элементы достижимы, противоречий нет |
| `bad_sv1_inconsistent` | Семантическая ошибка | СВ-1 фиксирует противоречие |
| `bad_sv1_grade_conflict` | Семантическая ошибка | СВ-1 фиксирует невыполнимый интервал оценки |
| `bad_sv2_simple_cycle` | Структурная ошибка | СВ-2 фиксирует цикл длины 2 |
| `bad_sv2_multi_type_cycle` | Структурная ошибка | СВ-2 фиксирует цикл через разнотипные рёбра |
| `bad_sv3_unreachable` | Операционная ошибка | СВ-3 фиксирует недостижимый элемент |
| `bad_sv3_unreachable_via_group` | Операционная ошибка | СВ-3 фиксирует недостижимость через цепочку группы и компетенции |
| `bad_sv3_dead_grade` | Операционная ошибка | СВ-3 фиксирует невыполнимый порог оценки |

Прогон полного набора подтверждает: на `happy_path` отчёт о верификации содержит нулевое число проблем по всем трём свойствам; на каждом `bad_sv*` обнаруживается дефект ожидаемого класса с правильным указанием затронутых элементов и правил. Файлы сценариев и таблица эталонных результатов хранятся в репозитории кода.

## 7.3. Стабилизация прототипа

В ходе интеграционного контура и ревью кода обнаружен и устранён ряд дефектов и недоработок.

| ID | Проблема | Способ устранения |
|---|---|---|
| TD2 | Отсутствие модульных тестов | Покрытие доведено до 84% общего и 82% по ядру |
| TD3 | Нет воспроизводимой среды развёртывания | Добавлен Docker Compose с тремя сервисами и проверками работоспособности |
| TD5 | Тип `date_restricted` опирался на Python-фильтр в обход SWRL | Переведён на enricher `CurrentTime` и сравнение в SWRL |
| TD6 | Отсутствовала проверка непротиворечивости при создании правила | Добавлен `consistent`-вызов с откатом и кодом ответа HTTP 409 |
| TD7 | Демо-данные были разрозненными | Сценарии собраны под общий модуль с эталонными ожиданиями |
| TD8 | Некорректные связи между компетенциями в OWL | Поправлен поиск целевой компетенции в обработчике запроса |
| TD9 | Комментарии TODO/FIXME в backend | Все упоминания выпилены или конвертированы в задачи |
| TD10 | Кэш не учитывал версию онтологии | Введена `ontology_version`, обход ключей через `SCAN`, синхронизация на старте |
| TD11 | Ответы `verify` и `explain` возвращались без явных схем | Введены пять Pydantic-моделей и подключение `response_model` |

Часть обнаруженных проблем намеренно оставлена за рамками практики: отсутствие RBAC (TD1), полный пересчёт выведенных фактов вместо инкрементального (TD4), вспомогательные функции кэша для multi-worker-режима (TD12). Каждая позиция сопровождена обоснованием в репозитории и относится к подсистемам, не пересекающимся с основным контуром управления правилами доступа.

## 7.4. Smoke-проверка пользовательского интерфейса

После каждого изменения серверного контура выполнялась ручная smoke-проверка симулятора (UC-7a, UC-7b, UC-7c) и редактора правил. Проверялась корректность отображения карты доступов после изменения фактов тестового обучающегося, согласованность редактора с серверной валидацией для всех типов политик, читаемость отчёта о верификации. Регрессий не зафиксировано.
