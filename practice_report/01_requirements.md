# 1. Анализ требований к программному комплексу

## 1.1. Существующие подходы в СДО

Анализ пяти ведущих платформ дистанционного обучения (Moodle, Canvas, Blackboard, Stepik, Open edX) показал, что во всех случаях правила доступа хранятся в реляционных БД (JSON-конфигурации, внешние ключи, проприетарные форматы) и интерпретируются процедурным кодом. Основная проблема заключется в том, что ни одна из платформ не выполняет формальную верификацию правил в процессе проектирования образовательных траекторий. Платформы документально предупреждают о риске циклических пререквизитов и недостижимых состояний, что провоцирует ситуации, когда учебный элемент ошибочно блокируется для обучающегося.

## 1.2. Стейкхолдеры

| Стейкхолдер            | Роль                                                                                  | Ключевая потребность                                                                                        |
| --------------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Методист                  | Создаёт правила доступа, проектирует траекторию | Понятный способ задать условие, проверка корректности до публикации |
| Обучающийся            | Потребитель учебного контента                                  | Прозрачность блокировок, отсутствие тупиков из-за ошибок методиста   |
| Администратор СДО | Управляет платформой, импорт и миграция курсов    | REST-интеграция, аудит набора правил при импорте                                          |
| Разработчик СДО     | Встраивает компонент в собственную платформу      | Платформо-независимый сервис со стандартным API                                         |

## 1.4. Функциональные требования

Восемь групп требований, выведенных из границ системы, пула типов правил и анализа стейкхолдеров.

| ID     | Группа                                  | Содержание                                                                                                                                                                                                                             |
| ------ | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ФТ-1 | Управление правилами       | CRUD-операции с правилами 9 типов на элементах любого уровня иерархии (активность, модуль, курс), активация и деактивация без удаления |
| ФТ-2 | Решение о доступе              | Вычисление решения для пары (обучающийся, элемент) на основе SWRL-правил и фактов в ABox с CWA-enforcement                                                                      |
| ФТ-3 | Верификация                        | Проверка непротиворечивости (СВ-1), обнаружение циклов (СВ-2) и недостижимых элементов (СВ-3) с возвратом структурированного отчёта     |
| ФТ-4 | Синхронизация с СДО          | Приём структуры курса и событий прогресса (просмотр, завершение, оценка, компетенция), пересчёт затронутых решений                               |
| ФТ-5 | Валидация на лету              | Предварительная проверка правила при создании, отклонение при нарушении свойств СВ-1 или СВ-2                                                                       |
| ФТ-6 | Трассировка вывода           | Список невыполненных условий для заблокированного элемента с типом, целевым элементом, требуемым и текущим значениями                      |
| ФТ-7 | Симулятор методиста         | Создание тестового обучающегося, изменение его фактов, отображение карты доступов с объяснением блокировок                                          |
| ФТ-8 | Агрегация завершённости | Каскадная фиксация завершения элементов-контейнеров по обязательным потомкам                                                                                                |

## 1.6. Сценарии использования

Выделены десять сценариев, покрывающих все восемь групп функциональных требований. Полная диаграмма приведена в Приложении А.

| UC-7b | Методист                             | Изменение фактов тестового обучающегося                        |
