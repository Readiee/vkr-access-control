# 2. Определение поддерживаемых типов правил доступа

## 2.1. Методика формирования пула

Состав поддерживаемых типов выводился из четырёх источников. Первый — анализ пяти СДО (раздел 1.1): фиксировалось, какие условия доступа реализованы в реальных платформах. Второй — матрица функциональных возможностей, сопоставляющая 20 практических сценариев с выразительной силой OWL+SWRL: SWRL-native, гибридная реализация (Python-предобработка + SWRL-сравнение), невозможность на уровне формализма. Третий — анализ литературы по адаптивным траекториям и стандартам e-learning. Четвёртый — обоснованность исключений: каждый тип, не вошедший в ядро, сопровождается причиной.

## 2.2. Состав ядра

Девять типов покрывают рыночный стандарт условий доступа в СДО и иллюстрируют две формы реализации: SWRL-native через прямое сопоставление атомов и гибридная схема с предобработкой ABox.

| № | Тип | Реализация | Покрытие СДО |
|---|---|---|---|
| 1 | `completion_required` | SWRL-native | пять из пяти |
| 2 | `grade_required` | SWRL + built-ins | пять из пяти |
| 3 | `viewed_required` | SWRL-native | Moodle, Blackboard |
| 4 | `competency_required` | SWRL + TransitiveProperty | Moodle |
| 5 | `date_restricted` | гибрид: enricher `CurrentTime` + SWRL-сравнение | четыре из пяти |
| 6 | `and_combination` | SWRL-native, конъюнкция в теле правила | Moodle, Blackboard |
| 7 | `or_combination` | SWRL-native, через `has_subpolicy` | Moodle, Blackboard |
| 8 | `group_restricted` | SWRL-native через `belongs_to_group` | Moodle, Canvas, Blackboard |
| 9 | `aggregate_required` | гибрид: enricher `AggregateFact` + SWRL-сравнение | Moodle, Blackboard, Open edX |

## 2.3. Обоснованные исключения

Часть возможностей была рассмотрена и не вошла в ядро.

- NOT-комбинация. Гипотеза открытого мира (OWA) формально несовместима с отрицанием как неудачей (NAF). Заменяющий архитектурный приём — запрет по умолчанию (default-deny) на уровне приложения.
- Ролевые ограничения (RBAC). Авторизация пользователей относится к подсистеме идентификации; разрабатываемая система управляет доступом на уровне контента.
- Адаптивная маршрутизация по диапазонам оценок. Эквивалентно `or_combination` из двух правил `grade_required` с граничными порогами; отдельный тип не вводится.
- Подсчёт попыток и время прохождения. Реализуемо по схеме гибридного типа, нишевые сценарии remediation. Вынесено в перспективы.
- Межкурсовые зависимости. Технически реализуемо в SWRL, выходит за рамки основного тезиса о верификации правил отдельного курса.

## 2.4. Верификационный слой

Верификационные свойства покрывают три ортогональных класса корректности набора правил курса.

| ID | Класс корректности | Что гарантирует | Инструмент |
|---|---|---|---|
| СВ-1 | Семантическая | Отсутствие логических противоречий между правилами | DL-резонер, проверка непротиворечивости |
| СВ-2 | Структурная | Граф зависимостей — ациклический | Графовый алгоритм |
| СВ-3 | Операционная | У каждого элемента есть достижимое состояние, в котором он доступен | Графовый анализ достижимости с проверкой выполнимости условий |

Свойства СВ-1, СВ-2 и СВ-3 формируют ядро верификации. Свойства оптимизационного класса (избыточность и поглощение правил) рассматриваются как направление дальнейшего расширения и в рамках практики не реализованы.
