# 4. Проектирование архитектуры и интерфейсов

## 4.1. Состав компонентов

Программный комплекс состоит из четырёх крупных компонентов, разнесённых по трём контейнерам.

| Компонент | Контейнер | Ответственность |
|---|---|---|
| База знаний | Backend | Хранение онтологии TBox + ABox, выполнение логического вывода через резонер Pellet, материализация выведенных фактов |
| Бэкенд-сервис | Backend | Приём REST-запросов, оркестрация конвейера логического вывода, верификация набора правил курса, синхронизация с внешней СДО |
| Кэш-слой | Cache | Хранение результатов решения о доступе и согласованности с версией онтологии |
| Веб-интерфейс | Frontend | Редактор правил, симулятор обучающегося, отчёт о верификации курса |

C4-контекст и C4-контейнеры приведены в Приложении В.

## 4.2. REST API

Прикладной интерфейс описан спецификацией OpenAPI 3.0. Точки входа сгруппированы в шесть роутеров по функциональным блокам.

| Роутер | Префикс | Назначение |
|---|---|---|
| `integration` | `/integration` | Импорт курса со структурой, синхронизация фактов прогресса |
| `policies` | `/policies` | CRUD-операции с политиками доступа, валидация на лету |
| `access` | `/access` | Решение о доступе, объяснение блокировки |
| `progress` | `/progress` | События прогресса обучающегося, восходящая агрегация завершённости |
| `verification` | `/verification` | Полная верификация курса по свойствам СВ-1, СВ-2, СВ-3 |
| `sandbox` | `/sandbox` | Песочница методиста: тестовый обучающийся, изменение фактов, карта доступов |

## 4.3. Схема данных

Контракт REST API описан 20 Pydantic-моделями. Доступ к ABox онтологии инкапсулирован в репозиториях по домену сущности.

| Репозиторий | Сущность ABox |
|---|---|
| `StudentRepository` | Обучающиеся, их группы, факты прогресса |
| `CourseRepository` | Курсы, модули, активности, иерархия `is_part_of` |
| `ProgressRepository` | События завершения, оценки, присвоение компетенций |
| `PolicyRepository` | Политики доступа всех типов и их подполитики |

## 4.4. Ключевые сценарии взаимодействия

В отчёте приведены два сценария: создание правила с валидацией на лету и проверка доступа обучающегося. Полный набор sequence-диаграмм (импорт курса, агрегация завершённости, конвейер логического вывода, обработка события прогресса) вынесен в Приложение В.

Создание правила доступа (UC-1, UC-3): запрос `POST /policies` → формирование политики во временной копии графа зависимостей → проверка на циклы (СВ-2) → добавление аксиом во временную копию онтологии → проверка непротиворечивости через Pellet (СВ-1) → при успехе фиксация в основной онтологии и сброс кэша. При нарушении любого из свойств возвращается HTTP 409 с указанием обнаруженной проблемы.

Проверка доступа (UC-4): запрос `GET /access/{student_id}/{element_id}` → обращение к кэшу → cache hit возвращает решение без логического вывода. При cache miss запускается конвейер: предобработка ABox (`CurrentTime`, агрегаты) → запуск Pellet → постобработка с применением запрета по умолчанию → запись результата в кэш с привязкой к версии онтологии. Ответ содержит решение `accessible` и список невыполненных условий.
