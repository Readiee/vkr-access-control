# 5. Реализация серверных компонентов прототипа

## 5.1. Структура backend

Серверная часть реализована на FastAPI. Код разделён на четыре слоя: маршрутизация (роутеры), бизнес-логика (сервисы), доступ к данным (репозитории) и инфраструктура (онтология, кэш, графовый анализ). Связи между сервисами разрешаются явным DI-графом, собранным в модуле `api/dependencies.py`. Сервис `OntologyCore` ограничен загрузкой онтологии и репозиториями; кэш и резонер инжектируются извне.

| Сервис | Ответственность |
|---|---|
| `PolicyService` | CRUD политик доступа всех типов с дискриминированной валидацией |
| `AccessService` | Решение о доступе, сборка списка невыполненных условий |
| `VerificationService` | Полная верификация курса по СВ-1, СВ-2, СВ-3 |
| `ReasoningOrchestrator` | Конвейер логического вывода: предобработка ABox, запуск Pellet, постобработка |
| `RollupService` | Восходящая агрегация завершённости по иерархии модулей |
| `ProgressService` | Приём событий прогресса, обновление ABox |
| `IntegrationService` | Импорт структуры курса и набора правил из внешней СДО |
| `SandboxService` | Песочница методиста, тестовый обучающийся |
| `OntologyCore` | Загрузка онтологии, управление world и сохранение |
| `CacheManager` | Работа с Redis, согласованность с версией онтологии |
| `GraphValidator` | Графовые алгоритмы цикл-анализа и достижимости |

## 5.2. Конвейер логического вывода

Реализован четырёхшаговый конвейер обработки запроса доступа. На вход поступает пара (обучающийся, элемент); на выход — решение и список невыполненных условий.

1. Предобработка ABox: материализация единственного экземпляра `CurrentTime` системным временем; вычисление `AggregateFact` по перечню целевых элементов каждой политики `aggregate_required`.
2. Запуск Pellet через Owlready2 в изолированном `world`. Применён патч маршалинга Jena → OWLAPI; передан таймаут.
3. Постобработка результата: запрет по умолчанию (`is_available_for` без выведенного `Permission` = доступ запрещён).
4. Запись в Redis с привязкой к версии онтологии.

После каждого изменения ABox или TBox выведенные факты `is_available_for` очищаются и конвейер запускается заново. Подход обусловлен монотонностью OWL: резонер не поддерживает откат выведенных следствий.

## 5.3. Алгоритмы верификации

Свойство СВ-1 (непротиворечивость) проверяется штатным сервисом резонера. В обработчике `PolicyService.create_policy` вызов `world.consistent()` обёрнут в обработку `OwlReadyInconsistentOntologyError`: при возбуждении исключения транзакция отменяется, ABox возвращается к состоянию до операции, клиенту возвращается HTTP 409 с обоснованием вывода.

Свойство СВ-2 (отсутствие циклов) проверяется графовым алгоритмом. Граф строится не на элементах напрямую, а через типозависимый split-node: каждый элемент представлен парой узлов «вход» и «выход» с типом ребра, отражающим характер зависимости (требование завершения, оценки, компетенции, членства в группе). Поверх графа применяется поиск сильно связных компонент (NetworkX, алгоритм Тарьяна). Любая компонента размера больше единицы интерпретируется как цикл; в ответ возвращается перечень элементов цикла и типы рёбер.

Свойство СВ-3 (достижимость элементов) проверяется обходом в обратную сторону: для каждого элемента курса строится множество предков по графу зависимостей и проверяется, существует ли непротиворечивая комбинация фактов (завершения, оценки, членства в группах), при которой все условия выполняются. При отрицательном результате элемент помечается недостижимым в отчёте о верификации.

## 5.4. Управление базой знаний

Загрузка онтологии и операции над ABox изолированы в сервисе `OntologyCore`. Каждая операция CRUD выполняется в основном `world`; для предварительной валидации (UC-3) используется временная копия графа и онтологии — транзакция отменяется, если проверка не прошла.

Версия онтологии вычисляется как `sha256` от mtime всех файлов TBox и SWRL-правил. Любое изменение онтологии меняет версию; кэш проверяет версию перед каждой выдачей значения. На старте приложения вызов `ensure_version_consistency` сверяет версии в Redis и в файлах онтологии; при расхождении кэш очищается обходом по шаблону через команду `SCAN`.

## 5.5. Объём кода и тестов

Серверная часть содержит 6 роутеров, 11 сервисов и 4 репозитория. Контракт REST API описан 20 Pydantic-моделями и одним перечислением. Покрытие автоматизированными тестами составляет 84% по проекту в целом и не менее 82% по ядру (сервисам логического вывода и верификации). Полный состав тестового стенда и сценарии описаны в разделе 7.
