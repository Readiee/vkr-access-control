# ЗАКЛЮЧЕНИЕ

> **Статус:** черновик. Структура — [SAT_PZ_REQUIREMENTS.md §4.6](../docs/SAT_PZ_REQUIREMENTS.md).

## Решённые задачи

В рамках работы решены шесть задач, поставленных во введении.

Анализ механизмов управления доступом в системах дистанционного обучения выполнен в главе 1. Проанализированы пять платформ (Moodle, Canvas LMS, Blackboard Learn, Stepik, Open edX), составлена сводная таблица возможностей и ограничений, выявлены семь общих проблем рынка от процедурного представления семантики правил до отсутствия машиночитаемой трассировки блокировки.

Разработка OWL-онтологии и каталога SWRL-правил выполнена в главах 2 и 3. Онтология содержит 18 классов, 16 объектных свойств и 13 атрибутов данных, объединяющих концепты образовательного домена и управления доступом. Каталог содержит 12 SWRL-правил с двухуровневой семантикой, покрывающих девять типов правил доступа.

Гибридная архитектура спроектирована в главе 2. Сформулированы три слоя архитектуры: слой предобработки на Python с инжекцией текущего времени и предвычисленных агрегатов (компенсация ограничений LIM1 и LIM2), резонер дескрипционной логики Pellet с правилами SWRL, слой постобработки в режиме default-deny (компенсация ограничения LIM5).

Модуль верификации реализован в главе 3. Сервис `VerificationService` объединяет проверку непротиворечивости через Pellet и графовый анализ зависимостей через NetworkX; покрыты три верифицируемых свойства СВ-1, СВ-2, СВ-3. Реализованы четыре алгоритма работы системы: построение графа зависимостей и обнаружение циклов (А1), оркестрация конвейера логического вывода (А2), восходящая агрегация завершённости (А3), проверка достижимости элементов (А4).

Экспериментальная оценка проведена в главе 4. Выполнены три эксперимента: обнаружение дефектов правил с F1 = 1.0 на 72 сценариях, корректность логического вывода с accuracy = 1.000 на 401 ячейке матрицы доступа, производительность с соответствием требованиям НФТ-1, НФТ-2, НФТ-3 на эталонной платформе.

Демонстрация совместимости с системами дистанционного обучения зафиксирована в главе 3 и приложении Ж. Составлен трёхслойный маппинг онтологии на пять платформ, реализован референсный адаптер интеграции с Moodle: импорт структуры курса в виде Python-кода, два PHP-плагина в виде спецификаций.

## Научные результаты

Получены три научных результата, образующих новизну работы.

Онтологическая модель управления доступом к образовательному контенту объединяет концепты образовательного домена и управления доступом. Существующие образовательные онтологии не формализуют управление доступом; существующие онтологии управления доступом не содержат образовательных концептов. Предложенная модель покрывает оба домена в едином формальном пространстве.

Гибридная архитектура SWRL и Python с режимом закрытого мира на уровне приложения позволяет описывать правила доступа в формальном пространстве с онтологией при сохранении возможности автоматического анализа. Архитектура компенсирует формальные ограничения OWL и SWRL стандартизованными слоями предобработки и постобработки, не нарушая семантику базового логического вывода.

Связка дескрипционной логики и графового анализа применена к верификации правил доступа в системах дистанционного обучения. Декомпозиция трёх верифицируемых свойств между двумя техниками расширяет работу Назыровой и соавторов (2023) с уровня учебного плана до уровня отдельных активностей и расширяет работы Коловского и соавторов (2007), Хуана и соавторов (2009) с XACML на нативную OWL-онтологию.

## Практические результаты

Разработан программный комплекс, реализующий методы работы. Стек развёртывается через Docker Compose как четыре контейнера: серверная часть на FastAPI с резонером Pellet и библиотекой Owlready2, клиентская часть на Vue.js, кэш Redis, отдельный демонстрационный экземпляр Moodle. Серверная часть реализует девять типов правил доступа в каталоге SWRL-правил, шесть HTTP-роутеров для десяти вариантов использования, три уровня тестирования.

Качество реализации подтверждено количественно: 117 автоматизированных тестов с покрытием 84% общего и не менее 82% по основному слою. Демонстрационный сценарий happy_path и семь негативных сценариев `bad_sv*` обеспечивают повторяемые регрессионные проверки на каждом изменении кода.

Адаптер интеграции с Moodle реализован в режиме single PDP по архитектуре XACML. Импорт структуры курса работает на демонстрационном экземпляре Moodle 4.3; применение правил и подписка на события прогресса спроектированы как спецификации PHP-плагинов с конкретными интерфейсами Moodle. Маппинг онтологии на пять платформ дистанционного обучения зафиксирован в приложении Ж и покрывает три слоя сопоставления: классы, отношения и типы правил.

## Рекомендации по использованию

Основной бенефициар разработанного комплекса — компания ООО «Дистех» как партнёр работы и заказчик при развитии собственной системы дистанционного обучения. Платформо-независимая архитектура комплекса позволяет встроить его в виде точки принятия решений в существующую инфраструктуру компании, обеспечив возможность формальной верификации правил доступа.

Комплекс может также применяться как референсная реализация для других разработчиков систем дистанционного обучения: онтология фиксирует формальный язык описания правил, общий для всего рынка; маппинг на пять платформ показывает, как этот язык соотносится с существующими механизмами; архитектура XACML с режимом single PDP даёт инженерное решение интеграции без перестройки целевой платформы.

Платформо-независимая онтология может рассматриваться как основа для отраслевой стандартизации формального описания правил доступа к образовательному контенту, аналогичной по роли стандарту xAPI для описания учебного опыта.

## Перспективы развития

Перспективы развития работы организованы четырьмя направлениями.

Расширение функциональности ядра системы. Из исключённых типов правил в перспективы вынесены межкурсовые зависимости (правило вида «завершение курса A открывает курс B»), адаптивная маршрутизация по диапазонам оценок (паттерн Mastery Paths), количество попыток прохождения и время, проведённое в активности. Каждый из перечисленных типов технически реализуется по тому же шаблону, что и существующие гибридные правила (предобработка на Python с инжекцией предвычисленных атрибутов в ABox), и не требует изменения архитектуры. Отдельная перспектива — гибридизация с Answer Set Programming для нативной поддержки отрицания через недоказуемость и инкрементальный логический вывод при росте размера ABox.

Расширение возможностей верификации. На ранних стадиях работы рассматривались два дополнительных свойства правил доступа: избыточность правил (правило поглощается другим правилом, и его удаление не меняет результата) и поглощение классов правил (правило для подмножества обучающихся поглощается правилом для надмножества). Свойства выведены из защищаемого объёма по причине того, что их синтаксическая реализация требует доказательства корректности через эквивалентности дескрипционной логики, не вписывающегося в срок работы. В перспективе обнаружение этих свойств реализуется через резонер на синтетических утверждениях вместо синтаксических эвристик. Дополнительно вынесена реализация прохода совместной проверки в алгоритме А4 для класса недостижимости через невыполнимость пересечения путей в графе зависимостей.

Завершение интеграции с Moodle. Импорт структуры курса реализован прототипом, применение правил и подписка на события прогресса спроектированы спецификациями PHP-плагинов. В перспективы развития вынесены: реализация custom availability condition plugin в директории `availability/condition/external_pdp/`, реализация плагина-наблюдателя событий с регистрацией подписок на три события Moodle, реализация reconciliation worker для периодической сверки состояний завершённости через Web Services, пилотное внедрение совместно с компанией-партнёром ООО «Дистех» с подготовкой акта о внедрении.

Расширение совместимости с другими системами дистанционного обучения. Маппинг онтологии на пять платформ зафиксирован в приложении Ж; в перспективы вынесены адаптеры для четырёх платформ помимо Moodle. Адаптер для Open edX наиболее прост после Moodle, поскольку Open edX написан на Python и поддерживает архитектуру плагинов. Адаптеры для Canvas LMS и Blackboard Learn реализуются через REST-интерфейсы платформ и стандарт LTI. Адаптер для Stepik имеет ограниченные возможности из-за закрытости и облачной природы платформы.
