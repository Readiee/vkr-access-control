# 3 РЕАЛИЗАЦИЯ ПРОГРАММНОГО КОМПЛЕКСА

> **Статус:** черновик. Структура и приоритеты по объёму — [SAT_PZ_REQUIREMENTS.md §4.4 + §20](../docs/SAT_PZ_REQUIREMENTS.md).

## 3.1 Технологический стек и обоснование выбора

Технологический стек разрабатываемого комплекса собран из шести компонентов: язык реализации Python, веб-фреймворк FastAPI, библиотека работы с онтологиями Owlready2, резонер дескрипционной логики Pellet, библиотека графового анализа NetworkX, кэш Redis. Клиентская часть построена на Vue.js с библиотекой компонентов PrimeVue.

Выбор Python обусловлен наличием единственной Python-библиотеки с поддержкой OWL и SWRL — Owlready2. Альтернативные библиотеки Apache Jena и OWL API реализованы на Java; их применение потребовало бы межъязыковых мостов между слоем предобработки (LIM1, LIM2 — инжекция фактов) и резонером, что усложняет архитектуру без функционального выигрыша. Библиотека RDFLib для Python поддерживает OWL, но не имеет средств работы со SWRL и логического вывода. Owlready2 связывает онтологические сущности с Python-объектами в едином адресном пространстве, что упрощает обмен данными между слоем предобработки на Python и резонером, запускаемым как Java-подпроцесс через JPype.

FastAPI выбран как фреймворк серверной части по совокупности характеристик: поддержка асинхронной обработки запросов, автоматическая генерация документации OpenAPI 3.0 (закрывает НФТ-9), типизация на основе аннотаций Python и валидация через Pydantic. Альтернатива Flask потребовала бы дополнительных пакетов для тех же возможностей.

Pellet выбран как DL-резонер по итогам сравнения (см. §1.7). Резонер интегрируется через Owlready2, поддерживает SWRL и встроенные функции swrlb, что обеспечивает требования LIM1 (инжекция текущего времени) и LIM2 (инжекция агрегатов) в гибридной архитектуре. Альтернативные резонеры HermiT и FaCT++ не поддерживают встроенные функции SWRL; Konclude не поддерживает SWRL целиком; гибрид HermiT с системой правил Drools требует двух машин логического вывода и усложняет обработку обновлений ABox.

NetworkX выбран как библиотека графового анализа. Объекты графа создаются непосредственно из ABox онтологии: вершины — индивиды учебных активностей и политик, рёбра — отношения зависимости. Алгоритмы поиска компонент сильной связности и обхода в глубину применяются для проверки свойства ацикличности (СВ-2). Альтернативная библиотека igraph быстрее на больших графах (порядка миллионов вершин), но для практических размеров курса (НФТ-4) NetworkX достаточен и не вносит дополнительных зависимостей.

Redis выбран как кэш результатов логического вывода. Обоснование — ограничение LIM4: задача логического вывода вычислительно дорога, а результаты стабильны между изменениями ABox. Стандартный паттерн кэширования с инвалидацией ключей при изменении состояния обучающегося или правил доступа закрывает требования НФТ-1 и НФТ-2 по времени отклика.

Клиентская часть на Vue.js с PrimeVue выбрана по знакомству разработчику; этот выбор не влияет на научный вклад работы. PrimeVue предоставляет готовый набор компонентов уровня корпоративного приложения, что сокращает объём имплементации интерфейса методиста.

Развёртывание комплекса выполняется через Docker Compose, что закрывает требование переносимости НФТ-12 на трёх операционных системах (Linux, macOS, Windows через WSL).

## 3.2 Реализация онтологии и логических правил

База знаний разрабатываемого комплекса состоит из терминологического компонента TBox, набора логических правил и инициализационного ABox демонстрационного курса. Сборка базы знаний автоматизирована тремя последовательно выполняемыми Python-скриптами в директории `code/onto/scripts/`, каждый из которых отвечает за свой слой: построение TBox, добавление правил и наполнение демонстрационным ABox. Структура и содержимое онтологии подробно описаны в главе 2 (§2.2); настоящий раздел сосредоточен на инженерных аспектах реализации.

### 3.2.1 Построение терминологического компонента

Скрипт `1_ontology_builder.py` создаёт онтологию через программный интерфейс Owlready2. Скрипт декларативно описывает иерархию из 18 классов, 16 объектных свойств и 13 атрибутов данных. Пять основных сущностей предметной области заданы корневыми классами `Course`, `LearningModule`, `LearningActivity`, `Student`, `AccessPolicy`. Иерархия политик образует подклассы для каждого из девяти типов правил §1.11: `CompletionRequiredPolicy`, `GradeRequiredPolicy`, `ViewedRequiredPolicy`, `CompetencyRequiredPolicy`, `DateRestrictedPolicy`, `AndCombinationPolicy`, `OrCombinationPolicy`, `GroupRestrictedPolicy`, `AggregateRequiredPolicy`.

Объектные свойства определяют отношения между индивидами: `belongs_to_course`, `contains_module`, `contains_activity`, `has_policy`, `belongs_to_group`, `requires_completion`, `acquired_competency` и другие. Свойство `is_subactivity_of` помечено как транзитивное — это позволяет резонеру автоматически выводить принадлежность активности корневому курсу через цепочку модулей произвольной глубины. Атрибуты данных хранят первичные значения: `is_mandatory` для активностей, `grade_threshold` для политик с условием по оценке, `start_date` и `end_date` для временных политик, `current_time_value` — точка инжекции текущего времени для гибридной реализации даты согласно LIM1.

После построения онтология сериализуется в формате RDF/XML в файл `code/onto/ontologies/access_control.owl`, что обеспечивает требование НФТ-10 о соответствии стандарту W3C. Сериализованная форма онтологии — единственный артефакт, передаваемый между слоем построения и серверной частью; отсутствие промежуточных форматов упрощает воспроизводимость.

### 3.2.2 Реализация логических правил

Скрипт `2_rules_setup.py` добавляет к онтологии 12 SWRL-правил, реализующих двухуровневую семантику доступа. Двухуровневая семантика разделяет вывод на два этапа: первый этап выводит факт `satisfies(student, policy)` для каждой пары обучающийся–политика, удовлетворяющей условиям политики; второй этап выводит факт `is_available_for(activity, student)` для каждой активности, для которой удовлетворены все назначенные ей политики. Разделение выводов на два уровня делает SWRL-правила компактными и позволяет переиспользовать предикат `satisfies` в проверке поглощения политик (см. §3.7.1).

Правила первого уровня — по одному на каждый из девяти типов политик. Правило для типа `completion_required` содержит конъюнкцию атомов в теле, проверяющих факт завершения целевой активности обучающимся; правило для `grade_required` использует встроенную функцию `swrlb:greaterThanOrEqual` для сравнения оценки с порогом. Правила для типов `and_combination` и `or_combination` опираются на свойство `has_subpolicy`, связывающее композитное правило с входящими в него подправилами. Правила для типов `date_restricted` и `aggregate_required` сравнивают значения, инжектированные слоем предобработки в индивиды `CurrentTime` и `AggregateFact` соответственно.

Правила второго уровня — три правила, выводящих `is_available_for` в зависимости от количества назначенных активности политик: ноль политик (доступно по умолчанию), одна политика (доступно при удовлетворении единственной политики), две политики (доступно при удовлетворении обеих). Поддержка большего числа политик обеспечена тем же шаблоном с сравнением через `owl:differentFrom` для разделения индивидов политик. Каждое SWRL-правило помечено атрибутом `swrl:label`, содержащим идентификатор шаблона, что упрощает диагностику и трассировку срабатывания правил при отладке.

### 3.2.3 Формирование демонстрационного ABox

Скрипт `3_seed_demo_data.py` подключает один из доступных сценариев в директории `code/onto/scenarios/`. Основной демонстрационный сценарий `happy_path.py` описывает курс с четырьмя обучающимися, тремя группами, четырьмя компетенциями, девятью политиками всех типов и 21 записью прогресса. Сценарий построен так, чтобы покрыть все девять типов правил и продемонстрировать корректное поведение системы в типичной педагогической ситуации: часть обучающихся успешно проходит курс, часть блокируется по разным типам условий.

Сценарий используется в трёх контекстах работы: смоук-проверка полного конвейера при разработке (см. §3.7.3), один из 72 сценариев в выборке эксперимента 1 главы 4, а также как демонстрационный набор данных в адаптере Moodle (§3.4.2). Полное описание ABox с листингом сценария, четырьмя обучающимися, тремя группами, политиками и записями прогресса приведено в приложении В.

Помимо `happy_path.py`, директория `scenarios/` содержит семь негативных сценариев `bad_sv*.py`, специально нарушающих одно из трёх верифицируемых свойств. Эти сценарии используются как ground truth для эксперимента 1: каждый сценарий содержит ровно один заранее известный дефект, и проверка состоит в обнаружении дефекта системой верификации.

## 3.3 Реализация серверной части системы

Серверная часть комплекса реализована на Python 3.11 в директории `code/backend/`. Программный интерфейс соответствует спецификации OpenAPI 3.0 версии 2.2.0, генерируемой автоматически из аннотаций FastAPI. Серверная часть взаимодействует с базой знаний через Owlready2, с резонером Pellet через JPype-мост и с кэшем Redis через стандартный клиент. Объём кода серверной части — около 4700 строк Python, распределённых по слоям API, сервисов, репозиториев и схем данных.

### 3.3.1 Слоистая архитектура серверной части

Серверная часть организована тремя слоями. Внешний слой API содержит шесть HTTP-роутеров FastAPI, отвечающих за приём и маршрутизацию запросов. Средний слой сервисов содержит бизнес-логику работы с правилами, прогрессом, доступом, верификацией и песочницей. Внутренний слой Core содержит работу с онтологией, кэшем и конвейером логического вывода. Зависимости направлены сверху вниз: роутеры зависят от сервисов, сервисы — от слоя Core, обратные зависимости отсутствуют.

Внедрение зависимостей выполнено через провайдеры FastAPI в модуле `api/dependencies.py`. Каждый сервис создаётся как одиночный экземпляр на время жизни приложения; роутеры получают экземпляры сервисов как параметры обработчиков, помеченные функцией `Depends`. Такая организация обеспечивает изолированное тестирование (каждый сервис подменяется фикстурой) и единую точку конфигурирования зависимостей.

### 3.3.2 Сервисный слой

Сервисный слой содержит восемь одиночных сервисов и три подпакета. Одиночные сервисы реализуют простые операции: `OntologyCore` инкапсулирует загрузку онтологии и Owlready2-сессии, `CacheManager` обёртывает Redis-клиент, `PolicyService` выполняет CRUD-операции над политиками, `ProgressService` принимает события прогресса, `RollupService` реализует алгоритм восходящей агрегации завершённости (А3 в §2.4.3), `SandboxService` управляет тестовыми обучающимися песочницы, `IntegrationService` принимает структуру курса и события из внешних систем дистанционного обучения, `GraphValidator` строит граф зависимостей и проверяет ацикличность алгоритмом А1.

Три подпакета группируют сервисы со сложной внутренней структурой. Подпакет `access` содержит сервис вычисления доступа `service.py` и вспомогательный модуль `_explanations.py`, формирующий человекочитаемые объяснения причин блокировки (требование ФТ-6). Подпакет `reasoning` содержит оркестратор конвейера логического вывода `orchestrator.py` и слой предобработки `_enricher.py`, инжектирующий текущее время и предвычисленные агрегаты в ABox согласно LIM1 и LIM2. Подпакет `verification` содержит сервис полной верификации `service.py` и модуль `_subsumption.py`, реализующий вспомогательные проверки поглощения политик (используется внутри сервиса для оптимизации выводов).

### 3.3.3 Контроллеры и маршрутизация запросов

Слой API содержит шесть роутеров FastAPI, каждый из которых отвечает за свою группу вариантов использования. Роутер `integration` (UC-10) принимает структуру курса и события прогресса от внешних систем. Роутер `policies` (UC-1, UC-2) обрабатывает создание, редактирование, удаление и активацию политик. Роутер `access` (UC-4, UC-9) возвращает решение о доступе и его объяснение. Роутер `progress` (UC-5) принимает отдельные события прогресса. Роутер `verification` (UC-6) запускает полную верификацию курса. Роутер `sandbox` (UC-7) управляет тестовыми обучающимися песочницы.

Маршрутизация в FastAPI организована через декораторы `@router.get`, `@router.post`, `@router.put`, `@router.delete`. Каждый обработчик объявляет тип входной модели и тип возвращаемого значения через аннотации Pydantic; FastAPI автоматически выполняет валидацию входных данных и сериализацию выходных. Спецификация OpenAPI генерируется из аннотаций без ручного описания, что закрывает требование НФТ-9 без затрат на сопровождение спецификации.

### 3.3.4 Репозитории и схемы данных

Слой репозиториев содержит модуль `ontology_repositories.py` с четырьмя классами доступа к данным: `StudentRepository`, `CourseRepository`, `ProgressRepository`, `PolicyRepository`. Каждый класс инкапсулирует операции выборки и модификации индивидов соответствующего OWL-класса через интерфейс Owlready2. Сервисы взаимодействуют с онтологией только через репозитории, что обеспечивает локализацию изменений модели данных в одном модуле.

Слой схем данных содержит модуль `schemas.py` с 17 Pydantic-моделями валидации. Модели описывают форматы входных и выходных данных программного интерфейса: `CourseStructure` для структуры импортируемого курса, `ProgressEvent` для события прогресса, пара `AccessRequest` и `AccessResponse` для запроса и ответа проверки доступа, тройка `PolicyCreate`, `PolicyUpdate`, `PolicyView` для операций с политиками, `VerificationReport` для отчёта верификации и другие. Pydantic выполняет автоматическую проверку типов, диапазонов значений и обязательности полей; нарушение схемы приводит к ответу 422 с детальным описанием ошибки.

### 3.3.5 Интеграция с резонером

Интеграция с резонером Pellet выполнена через библиотеку Owlready2, которая поднимает Java-подпроцесс и взаимодействует с ним через JPype-мост. Резонер запускается в изолированном «мире» Owlready2 — отдельном пространстве имён, в которое загружается копия онтологии перед вызовом резонера. Изоляция гарантирует, что выводы резонера не загрязняют рабочую онтологию аксиомами, которые могут быть выведены в одной комбинации фактов и недопустимы в другой.

Стандартная сборка Owlready2 включает версию резонера, использующую устаревший API Apache Jena. Для совместимости со SWRL и встроенными функциями требуется версия резонера, использующая OWL API. В рамках разработки применён патч библиотеки Owlready2, заменяющий вызовы Jena на эквивалентные вызовы OWL API; патч локализован в одном модуле и не затрагивает остальные части библиотеки.

Вызов резонера ограничен таймаутом, заданным параметром НФТ-3 (по умолчанию 10 секунд). При превышении таймаута оркестратор конвейера возвращает ранее кэшированный результат, если он есть, или ошибку 503 с описанием ситуации. Поведение при таймауте соответствует требованию ФТ-2.4.

### 3.3.6 Кэширование результатов вывода

Кэш реализован поверх Redis. Подключение к Redis-серверу выполняется через переменную окружения `REDIS_URL`, что упрощает развёртывание в разных средах: локальная разработка использует контейнер Redis из Docker Compose, экспериментальная среда главы 4 — отдельный экземпляр.

Ключи кэша формируются как пара (идентификатор обучающегося, идентификатор активности), значения — сериализованный результат проверки доступа: булев флаг доступности, список невыполненных условий и временная метка вычисления. Стратегия инвалидации соответствует алгоритму §2.7.2 главы 2: при изменении состояния обучающегося (новое событие прогресса) инвалидируются ключи только этого обучающегося; при изменении правил доступа активности инвалидируются ключи только этой активности и её потомков по графу зависимостей; при изменении версии онтологии инвалидируется весь кэш. Гранулярная инвалидация снижает частоту полных пересчётов и поддерживает требования НФТ-1 и НФТ-2 по времени отклика на запросы доступа.

## 3.4 Интеграционный адаптер на примере Moodle

Адаптер интеграции с Moodle реализует режим Б — single PDP, описанный в §2.6. В этом режиме источником истины правил доступа выступает онтология разрабатываемого комплекса; Moodle является чистой точкой применения (PEP), делегирующей решение о доступе нашей точке принятия решений (PDP) через программный интерфейс. Адаптер реализован двумя частями: Python-часть выполняет первоначальный импорт структуры курса и составляет источник атрибутов в терминологии XACML; PHP-часть — два плагина Moodle, обеспечивающих доставку событий прогресса и применение правил при показе элемента.

Импорт структуры курса реализован прототипом и проверен на демонстрационном экземпляре Moodle 4.3. Применение правил и подписка на события спроектированы как спецификации PHP-плагинов с конкретными Moodle API; полная реализация плагинов и пилотное внедрение совместно с компанией-партнёром вынесены в перспективы развития (см. заключение).

### 3.4.2 Импорт структуры курса

Импорт выполняется скриптом `MoodleCourseImporter.import_course(course_shortname)` через программный интерфейс Web Services Moodle. Используются семь endpoint-ов: `core_course_get_courses_by_field` для получения метаданных курса по идентификатору shortname, `core_course_get_contents` для получения секций и активностей, `core_enrol_get_enrolled_users` для списка обучающихся, `core_group_get_course_groups` и `core_group_get_group_members` для групп и членства, `gradereport_user_get_grade_items` для исторических оценок и опциональный `core_competency_list_course_competencies` для компетенций курса.

Сопоставление сущностей Moodle и классов TBox разрабатываемой онтологии приведено в таблице 3.2. Полная версия таблицы с полями-связями для каждой сущности — приложение Ж.

Таблица 3.2 — Сокращённый маппинг сущностей Moodle на классы TBox

| Сущность Moodle | Класс TBox | Идентификатор индивида |
|---|---|---|
| `course` | `Course` | `course_<shortname>` |
| `section` | `LearningModule` | `module_<id>` |
| `course_module` | `LearningActivity` (подкласс по `modname`) | `activity_<id>` |

Алгоритм оркестратора импорта приведён ниже в семи шагах. На первом шаге адаптер получает метаданные курса по shortname и формирует индивид `Course`. На втором шаге обходит секции и активности, создавая `LearningModule` и `LearningActivity` через переводчики `translators.py`. На третьем шаге собирает структуру курса в формат Pydantic-модели `CourseSyncPayload` и отправляет её через единый запрос `POST /api/v1/courses/{id}/sync` нашей серверной части — серверная часть создаёт индивидов в ABox и автоматически запускает полную верификацию (UC-6). На четвёртом шаге адаптер получает обучающихся, группы и членство в группах, передавая их через специализированные endpoint-ы. На пятом шаге адаптер обходит gradebook и преобразует исторические оценки в события прогресса, отправляя их через `POST /api/v1/events/progress`. На шестом шаге выводит статистику импорта: количество секций, активностей, обучающихся, групп, событий прогресса и затраченное время. Седьмой шаг — серверная часть запускает первый цикл логического вывода и формирует кэш доступов.

Существующие availability conditions Moodle при импорте игнорируются. Это часть выбора режима Б: источник истины правил — наша онтология, и попытка интерпретировать локальный формат Moodle одновременно с собственным форматом правил создавала бы две точки истины с задачей синхронизации. Если в импортируемом курсе уже заданы availability conditions средствами Moodle, методист переописывает их через интерфейс разрабатываемой системы.

Подписка на события прогресса реализована плагином-наблюдателем `local_external_pdp/observer.php` (директория `code/integrations/moodle/php/event_observer.php` — спецификация). Плагин подписан на три события Moodle, перечисленные в таблице 3.3.

Таблица 3.3 — События Moodle, на которые подписан наблюдатель

| Событие | Когда генерируется |
|---|---|
| `\core\event\course_module_completion_updated` | обучающийся или система отметили активность как завершённую |
| `\mod_quiz\event\attempt_submitted` | сохранена попытка теста с финальной оценкой |

Надёжность доставки обеспечена паттерном at-least-once: плагин выполняет HTTP-вызов с таймаутом 2 секунды и при неуспехе сохраняет полезную нагрузку в локальный буфер Moodle, который периодически опрашивается reconciliation worker-ом для повторной отправки. Идемпотентность гарантируется upsert-семантикой `ProgressRepository`: повторный приём того же события не создаёт дубликата, а обновляет существующую запись. Полная реализация PHP-плагина (около 50 строк рабочего кода с регистрацией в `db/events.php`, локализацией строк интерфейса и юнит-тестами) отнесена в перспективы развития.

Последовательность взаимодействия Moodle и серверной части при наступлении события прогресса показана на sequence-диаграмме [diagrams/uml/sequence-moodle-event-import.mmd](../diagrams/uml/sequence-moodle-event-import.mmd) (рисунок 3.1).

### 3.4.4 Применение правил при показе элемента

Применение правил реализовано custom availability condition plugin в директории Moodle `availability/condition/external_pdp/`. Стандартный механизм Moodle предусматривает вызов метода `condition::is_available($cm, $userid)` плагина при каждом рендере страницы курса для каждой видимой активности. Плагин делегирует решение нашей системе через программный интерфейс.

Реализация метода `is_available` (спецификация в файле `code/integrations/moodle/php/availability_condition.php`) выполняет HTTP-вызов `GET /api/v1/access/student/student_{userid}/element/activity_{cmid}` с таймаутом 200 миллисекунд. Таймаут согласован с требованием НФТ-1: 50 миллисекунд для попадания в кэш плюс запас на сетевые накладные расходы. Тело ответа содержит булев флаг `accessible` и список `unmet_conditions` для случая отказа. При флаге `false` плагин формирует объяснение из списка невыполненных условий и возвращает его в локализованном виде через стандартный механизм Moodle.

При недоступности нашей системы (превышение таймаута, сетевая ошибка, статус ответа отличный от 200) плагин блокирует элемент. Эта стратегия (fail-close) соответствует принципу минимальных привилегий, описанному в стандарте NIST SP 800-162. Альтернативная стратегия fail-open снимала бы блокировку при сбое, но это нарушало бы гарантии безопасности доступа: обучающийся получал бы доступ к контенту, который должен быть скрыт, при отказе системы доступа.

Кэширование на стороне Moodle не реализовано. Наша серверная часть уже кэширует результаты решений в Redis с гранулярной инвалидацией; дублирование кэша на стороне Moodle усложнило бы инвалидацию (нужно было бы синхронизировать инвалидацию двух кэшей при каждом событии прогресса) без выигрыша по времени отклика.

Последовательность взаимодействия при рендере страницы курса показана на sequence-диаграмме [diagrams/uml/sequence-moodle-availability-check.mmd](../diagrams/uml/sequence-moodle-availability-check.mmd) (рисунок 3.2). Полная реализация PHP-плагина (около 100 строк рабочего кода с локализацией, юнит-тестами и формой настройки соединения) вынесена в перспективы.

### 3.4.5 Ограничения выбранного режима интеграции

Режим Б накладывает пять ограничений, существенных при практическом внедрении.

Сетевая задержка на каждом рендере страницы курса. На страницу с N видимыми активностями Moodle выполняет N HTTP-вызовов к нашей системе. При попадании в кэш каждый вызов укладывается в требование НФТ-1 (50 миллисекунд); при типичном N=20 совокупная задержка составляет порядка 30 миллисекунд и не влияет на пользовательский опыт. При промахе кэша (cold start пользователя) задержка может достигать 2 секунд (НФТ-2). Это происходит один раз при первом обращении пользователя в сессии и компенсируется прогревом кэша при входе.

Поведение при недоступности нашей системы. Плагин блокирует элемент по принципу fail-close. Безопасный выбор по умолчанию требует мониторинга доступности: длительный отказ нашей системы блокирует учебный процесс целиком.

Согласованность прогресса в реальном времени. Между событием Moodle (например, отметкой о завершении активности) и инвалидацией нашего кэша проходит время порядка 100–500 миллисекунд: HTTP-roundtrip плюс работа конвейера логического вывода. На практике это не критично, поскольку обучающийся не «обгоняет» резонер при последовательном прохождении активностей.

Гарантии доставки событий. Плагин-наблюдатель использует at-least-once с локальным retry-буфером в Moodle. Идемпотентность нашего эндпоинта обеспечена upsert-семантикой репозитория прогресса: повторный приём события не приводит к двойному учёту.

Периодическая сверка состояний. Reconciliation worker, опрашивающий `core_completion_get_activities_completion_status` Moodle и сравнивающий результат с ABox нашей системы, обнаруживает и компенсирует пропущенные события. Реализация worker-а вынесена в перспективы развития.

## 3.5 Реализация клиентской части системы

Клиентская часть реализована на Vue.js 3 с библиотекой компонентов PrimeVue в директории `code/frontend/`. Структура клиентской части включает семь функциональных компонентов, три представления, два хранилища состояния на Pinia и набор композаблов с переиспользуемой логикой. Компоненты реализуют интерфейс методиста и сценарии вариантов использования UC-1, UC-2, UC-3, UC-6, UC-7a, UC-7b, UC-7c, UC-9.

Представление `Dashboard` (UC-1, UC-2) обеспечивает редактирование политик доступа: компонент `PolicyWorkspace` отображает список правил активности, компонент `PolicyRuleCard` — карточку отдельной политики с редактируемыми параметрами, компонент `CompositePolicyEditor` — редактор композитных правил `and_combination` и `or_combination` с поддержкой произвольной вложенности.

Представление `Sandbox` (UC-7a, UC-7b, UC-7c) реализует песочницу методиста: компонент `SandboxInspector` показывает карту доступов тестового обучающегося со списком всех элементов курса и отметкой их доступности, компонент `BlockingExplanation` (UC-9) разворачивает причины блокировки выбранного элемента в виде списка невыполненных условий.

Представление `VerificationReport` (UC-6) отображает результат полной верификации курса: список обнаруженных проблем по трём свойствам СВ-1, СВ-2, СВ-3, с группировкой по типу свойства и развёрнутыми объяснениями. Компонент `JustificationTreeView` показывает обоснования вывода резонера в виде дерева, что позволяет методисту проследить, какие именно аксиомы и правила привели к обнаружению проблемы.

Хранилища состояния `ontology` и `sandbox` на Pinia централизуют работу с серверной частью: они инкапсулируют HTTP-вызовы к API через axios-клиент в `code/frontend/src/api/`, кэшируют ответы и обеспечивают реактивное обновление компонентов при изменении состояния. Композаблы выделяют переиспользуемую логику: форматирование сообщений об ошибках, обработку асинхронных операций с индикаторами загрузки, локализацию строк интерфейса.

## 3.6 Развёртывание программного комплекса

Развёртывание выполняется через Docker Compose: файл `code/docker-compose.yml` описывает три сервиса основного стека. Сервис `redis` использует официальный образ Redis 7 без дополнительной настройки. Сервис `backend` собирается из Dockerfile, включающего Python-зависимости из `requirements.txt` и Java Runtime Environment, требуемое для запуска резонера Pellet через JPype. Сервис `frontend` собирает клиентское приложение Vite-сборкой и обслуживает статические файлы через nginx с прокси-проксированием обращений к API на сервис backend.

Все три сервиса объявляют health-checks: backend проверяется через эндпоинт `GET /` корневого FastAPI-приложения, frontend — через стандартную проверку nginx, redis — через команду `redis-cli ping`. Зависимости запуска заданы директивой `depends_on` с условием `service_healthy`, что обеспечивает корректный порядок старта.

Демонстрационный Moodle для PoC адаптера запускается отдельным compose-файлом `code/integrations/moodle/docker-compose.moodle.yml` и не входит в основной стек развёртывания. Разделение продиктовано тем, что Moodle нужен только для воспроизведения экспериментов главы 4 и для иллюстрации интеграции, а в продуктивной среде заказчика подключение выполняется к существующему экземпляру Moodle через программный интерфейс.

## 3.7 Тестирование

Тестирование разрабатываемого комплекса выполняется на трёх уровнях: модульные тесты (unit) проверяют корректность отдельных функций и методов, интеграционные тесты (integration) проверяют взаимодействие компонентов, сценарные тесты (scenarios) проверяют поведение системы на типовых сценариях работы. Тестирование подтверждает корректность реализации относительно спецификации; эффективность алгоритмов на выборке количественно оценивается в главе 4 экспериментами 1–3.

### 3.7.1 Методология тестирования

Тестовый фреймворк — pytest. Структура тестов в `code/backend/src/tests/` повторяет структуру кода: каждому подпакету сервисного слоя соответствует директория тестов. Модульные тесты используют фикстуры pytest для подмены зависимостей: например, тесты сервиса `AccessService` получают подменённый `OntologyCore` с заранее загруженной онтологией без обращения к реальному файлу.

Интеграционные тесты в директории `tests/integration/` запускают полный конвейер логического вывода с резонером Pellet. Каждый интеграционный тест начинается с создания изолированного «мира» Owlready2 через метод `setUp`: в фикстуру загружается копия онтологии, что предотвращает влияние тестов друг на друга. После выполнения теста изолированный мир очищается, обеспечивая воспроизводимость.

Сценарные тесты построены поверх сценариев в `code/onto/scenarios/`: каждый сценарий — Python-файл, описывающий конфигурацию ABox, ожидаемое поведение системы и список проверок. Тесты прогоняют сценарий через сервис, сравнивают выводы с ожидаемыми и формируют отчёт. Демонстрационный сценарий `happy_path` используется одновременно в сценарных тестах (как смоук-проверка) и в выборке эксперимента 1 главы 4 (как один из положительных случаев), что обеспечивает преемственность между уровнями проверки.

### 3.7.2 Покрытие тестами

Покрытие тестами измеряется средствами `pytest-cov`. Текущие показатели — 117 тестов с покрытием 84% по всему коду серверной части и не менее 82% по слою Core, отвечающему за работу с онтологией и резонером. Эти значения превышают порог НФТ-11 (60% общего покрытия) и подтверждают пригодность реализации к экспериментальной оценке главы 4.

Тесты слоя Core имеют приоритет покрытия из-за центральной роли в корректности всей системы: ошибка в сервисе вычисления доступа `AccessService` или в оркестраторе конвейера `ReasoningOrchestrator` приводит к некорректным выводам, неотличимым от ошибки в правилах. Тесты слоя API имеют менее жёсткое требование, поскольку основная логика делегируется сервисам; тесты роутеров проверяют корректную маршрутизацию, валидацию входных данных и сериализацию ответов.

### 3.7.3 Иллюстративный сквозной прогон системы

Демонстрационный сценарий `happy_path` используется как один комплексный тестовый вход для проверки полного конвейера работы: импорт структуры курса, наполнение ABox, запуск логического вывода, проверка доступов, восходящая агрегация завершённости. Полное описание ABox сценария приведено в приложении В.

Прогон сценария `happy_path` выполняет последовательность из четырёх шагов. На первом шаге сервис `IntegrationService` принимает структуру курса с четырьмя обучающимися и набором политик. На втором шаге сервис `OntologyCore` загружает онтологию, репозитории создают индивидов в ABox, ProgressService регистрирует 21 событие прогресса. На третьем шаге `ReasoningOrchestrator` запускает конвейер: слой предобработки `_enricher.py` инжектирует текущее время и предвычисленные агрегаты, резонер Pellet выполняет логический вывод, слой постобработки применяет режим default-deny согласно LIM5. На четвёртом шаге сервис `AccessService` возвращает решения для всех пар обучающийся–активность; результаты сравниваются с ожидаемыми из спецификации сценария.

Сквозной прогон не доказывает корректность системы на произвольных сценариях; он подтверждает работоспособность системы на одном эталонном входе и применяется как смоук-проверка на каждом изменении кода. Количественная оценка корректности на выборке выполняется в эксперименте 1 главы 4.

## 3.8 Выводы по главе 3

В главе изложена реализация программного комплекса по проекту главы 2. Реализован полный технологический стек: Python с библиотекой Owlready2 и резонером Pellet, веб-фреймворк FastAPI, графовая библиотека NetworkX, кэш Redis, клиентский фреймворк Vue.js с библиотекой PrimeVue. Технологический стек охвачен Docker Compose для воспроизводимого развёртывания на трёх операционных системах согласно требованию НФТ-12.

База знаний включает терминологический компонент из 18 классов, 16 объектных свойств и 13 атрибутов данных, 12 SWRL-правил с двухуровневой семантикой и демонстрационный ABox со сквозным сценарием happy_path и семью негативными сценариями. Серверная часть реализована в трёх слоях: API, сервисы, Core. Сервисный слой содержит восемь одиночных сервисов и три подпакета (access, reasoning, verification). Программный интерфейс описан спецификацией OpenAPI 3.0 версии 2.2.0. Клиентская часть реализует семь компонентов и три представления, покрывающих восемь вариантов использования из десяти.

Адаптер интеграции с Moodle реализован прототипом импорта структуры курса и применения правил при показе элемента. PHP-плагины событийной шины и точки применения реализованы как спецификации; их полная имплементация и пилотное внедрение совместно с компанией-партнёром вынесены в перспективы развития согласно решению о сокращении объёма работы.

Качество реализации подтверждается количественными показателями тестирования: 117 тестов с покрытием 84% общего и не менее 82% по слою Core, что превышает порог НФТ-11. Все тесты проходят на момент завершения работы по реализации, что подтверждает готовность системы к экспериментальной оценке. Количественная оценка корректности и производительности на выборках сценариев и наборах данных приведена в главе 4.
