## 2.1 Архитектура разработанной системы

За основу взята архитектурная нотация C4 [@N17-c4model-ИСТОЧНИК: проверить]. Разработанная система состоит из четырёх контейнеров с трёхслойной внутренней структурой: внешний слой принимает запросы, средний инкапсулирует прикладную логику обработки правил, ядро объединяет онтологию OWL 2 DL и логический вывод на резонере Pellet.

## 2.2 Онтологическая модель предметной области

Онтологическая модель покрывает все типы правил доступа из раздела 1.12 и три верифицируемых свойства из раздела 1.8. За базовый формализм взят OWL 2 DL, расширенный правилами SWRL: онтологический слой обеспечивает классы, иерархию и аксиомы предметной области (терминологический компонент, далее TBox), отдельный слой утверждений (ABox) хранит факты конкретного курса, а SWRL добавляет правила вывода между фактами, которые в DL без расширения невыразимы.

### 2.2.1 Структура TBox

Структура TBox показана на рисунке 2.10. Компонент содержит классы предметной области, организованные в иерархию через `owl:subClassOf`, и набор свойств, связывающих экземпляры классов между собой и с базовыми типами данных.

Основные классы TBox и их назначение приведены в таблице 2.1.

Таблица 2.1 – Основные классы TBox

| Класс / группа | Назначение |
| --- | --- |
| `User` | Суперкласс пользователей; подклассы: `Student`, `Methodologist`, `Instructor` |
| `Group` | Группа обучающихся; связь через `belongs_to_group` |
| `Course`, `Module`, `Activity` | Трёхуровневая иерархия курса; подклассы `Activity` по типу: лекция, тест, задание, практическая работа |
| `ProgressRecord` | Прогресс обучающегося: активность, статус (`viewed / completed / failed / passed`), оценка |
| `Competency` | Компетенция; иерархия через транзитивное `is_subcompetency_of`; связь с активностями через `assesses` |
| `AccessPolicy` | Правило доступа в реифицированной форме: тип, порог, автор, флаг активности |
| `CurrentTime`, `AggregateFact` | Вспомогательные классы для гибридной архитектуры (раздел 2.3); пересоздаются перед каждым запуском вывода |

Среди свойств выделяются `satisfies` и `is_available_for`: первое связывает обучающегося с политикой и означает «условие политики выполнено», второе связывает элемент курса с обучающимся и означает «элемент доступен». Аксиомы TBox задают четыре класса ограничений: попарную дизъюнктность корневых классов, функциональность ключевых свойств, транзитивность `is_subcompetency_of` и инверсные пары для отношений вроде `has_module` и `is_contained_in_course`.

### 2.2.2 Каталог логических правил

Каталог логических правил организован в две ступени. На первой ступени для каждого из поддерживаемых типов правил доступа задан собственный SWRL-шаблон, выводящий промежуточное свойство `satisfies(student, policy)`. Краткое содержание шаблонов первой ступени приведено в таблице 2.2; полные тексты с биндингами переменных находятся в Приложении А.

Таблица 2.2 – Сводка шаблонов SWRL первой ступени

| № | Тип правила                        | Что требует от обучающегося                                                                                |
| -- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 1  | Завершение элемента        | Запись прогресса со статусом «завершено» по целевому элементу                |
| 2  | Оценка элемента                | Оценка в записи прогресса не ниже порога политики                                        |
| 3  | Просмотр элемента            | Запись прогресса со статусом «просмотрено» по целевому элементу            |
| 4  | Получение компетенции    | Владение требуемой компетенцией с учётом иерархического наследования |
| 5  | Временное окно                  | Текущее время попадает в интервал политики                                                    |
| 6  | Конъюнкция условий          | Удовлетворены все подполитики                                                                           |
| 7  | Дизъюнкция условий          | Удовлетворена хотя бы одна подполитика                                                           |
| 8  | Принадлежность к группе | Обучающийся состоит в требуемой группе                                                           |
| 9  | Агрегат по элементам       | Среднее, сумма или количество оценок выше порога                                          |

Мета-правило второй ступени переводит `satisfies` в `is_available_for(element, student)` для активных политик на элементе. Двухуровневое разделение даёт три практических следствия. Семантика логических комбинаций AND и OR выражается через свойство `has_subpolicy` без дублирования тел атомарных шаблонов. Знание о конкретных типах правил инкапсулировано в шаблонах первой ступени и не проникает в мета-правило. Трассировка блокировки в случае отказа опирается на единый список выведённых и невыведённых `satisfies` для каждой пары обучающийся–политика. Аналогичное разделение используется в архитектуре XACML, где проверка правила и принятие политики разнесены на отдельные стадии оценки.

Два шаблона работают по гибридному паттерну, обходя ограничения SWRL на текущее время и агрегатные функции. Шаблон ограничения по времени сравнивает значения `valid_from` и `valid_until` политики со значением в индивиде `current_time_ind` класса `CurrentTime`. Шаблон агрегатного требования сравнивает порог политики со значением `computed_value` из индивида `AggregateFact`. Оба индивида пересоздаются слоем предобработки перед каждым запуском логического вывода (раздел 2.3.1).

Каталог дополнен вспомогательным правилом H-1: если обучающийся владеет подкомпетенцией, транзитивная иерархия `is_subcompetency_of` переносит владение на родительскую компетенцию. Без этого правила транзитивность `is_subcompetency_of` не распространяется на свойство `has_competency` (это разные свойства), и шаблон требования компетенции не сработает.

### 2.2.3 Структура утверждений на демонстрационном курсе

Демонстрационный курс по основам Python используется как сквозной пример во всех главах работы. Он состоит из четырёх обязательных модулей с активностями разных типов плюс нескольких дополнительных активностей. В курсе участвуют четыре обучающихся с разными профилями прохождения, распределённых по трём группам; компетенциями допускается владеть вне курса.

Девять политик доступа на демонстрационном курсе покрывают все поддерживаемые типы правил. На одном элементе намеренно размещена ровно одна политика, чтобы при анализе графа зависимостей не возникало пересечения двух условий (намеренные пересечения для проверки верифицируемых свойств вынесены в отдельные негативные мини-ABox). Записи прогресса подобраны так, чтобы каждый шаблон логических правил имел в матрице вывода хотя бы одну строку с положительным и одну с отрицательным результатом.

Полный листинг политик с их атрибутами, набор записей прогресса, иерархия компетенций и ожидаемая матрица доступа `is_available_for` для каждого обучающегося приведены в Приложении В.

## 2.3 Гибридная архитектура логического вывода

Гибридная архитектура логического вывода объединяет OWL+SWRL логический вывод с Python-слоями предобработки и постобработки. Без расширения OWL+SWRL не покрывают всех потребностей задачи управления доступом: язык SWRL не имеет встроенных функций для текущего времени и агрегатных вычислений по множеству фактов, а гипотеза открытого мира OWL несовместима с семантикой запрета по умолчанию.

На рисунке 2.12 показан конвейер логического вывода. Запрос на проверку доступа или на верификацию курса проходит четыре стадии: предобработку фактов, логический вывод, постобработку с применением гипотезы закрытого мира, материализацию результата в кэш.

### 2.3.1 Слой предобработки фактов

Слой предобработки решает два формальных ограничения языка SWRL: отсутствие функции текущего времени и отсутствие агрегатных функций по множеству фактов. Решение в обоих случаях устроено одинаково: значения, которые SWRL не умеет вычислять, размещаются в фактах онтологии заранее, а сам SWRL сравнивает уже размещённые значения с пороговыми и интервальными ограничениями политик.

Для текущего времени слой предобработки перед каждым запуском логического вывода удаляет существующий индивид класса `CurrentTime` и создаёт новый со значением текущего времени системы. SWRL-шаблон ограничения по времени берёт это значение из единственного индивида и сравнивает с границами `valid_from` и `valid_until` политики.

Для агрегатных требований слой предобработки перед каждым запуском логического вывода создаёт семейство индивидов класса `AggregateFact`: по одному на каждую пару обучающийся–политика типа «агрегатное требование». В каждом индивиде размещается значение `computed_value`, рассчитанное Python-слоем по формуле политики (среднее, сумма или количество оценок обучающегося по элементам политики). SWRL-шаблон агрегатного требования сравнивает порог политики с заранее посчитанным значением.

Единый паттерн (размещение предвычисленного значения в индивиде онтологии) обходит оба ограничения SWRL одним архитектурным приёмом. Альтернативы вроде ASP или F-Logic поддерживают и текущее время, и агрегации в самом языке, но требуют отказа от стандарта W3C и от совместимости с литературой по онтологическому управлению доступом.

### 2.3.2 Слой постобработки и применения гипотезы закрытого мира

В OWL действует гипотеза открытого мира: отсутствие выведенного факта не равносильно его ложности. В задаче управления доступом нужна противоположная семантика: запрет по умолчанию по принципу наименьших привилегий. На уровне онтологии такое поведение невыразимо: SWRL не поддерживает отрицание через закрытый мир, а OWL не выводит отрицательный факт из отсутствия положительного.

Слой постобработки применяет гипотезу закрытого мира непосредственно к результатам логического вывода. Для каждой пары обучающийся–элемент проверяется, выведен ли факт `is_available_for(element, student)`; если нет — доступ запрещён по умолчанию.

Кроме запрета по умолчанию слой постобработки предусматривает каскадную блокировку по иерархии. Если методист ограничил доступ к модулю (например, поставил требование оценки за тест прошлого модуля), то все активности внутри ограниченного модуля автоматически наследуют отказ, не требуя отдельных политик на каждой активности. Каскад работает по иерархии «курс → модуль → активность»: сначала рассчитывается доступ для элемента, затем проверяются все родительские контейнеры; при отказе на любом уровне отказ распространяется на вложенные элементы. Это стандартный для систем управления доступом паттерн «deny-overrides».

### 2.4.1 Алгоритм обнаружения циклов в графе зависимостей

Алгоритм обнаружения циклов в графе зависимостей обеспечивает проверку свойства ацикличности из раздела 1.8. Цикл возникает, когда условие доступа к одному элементу прямо или косвенно зависит от состояния другого, который в свою очередь требует первого. Без обнаружения элементы взаимно блокируются навсегда.

Прямое построение графа «узел — элемент курса, ребро — политика-зависимость» работает только для атомарных типов завершения. Правила доступа смешивают два разных события: получение доступа к элементу и его завершение. Например, политика «требование завершения» означает «доступ к X разрешён, если завершён Y»; зависимость идёт от завершения одного к доступу другого, и разные типы политик создают разные виды рёбер.

Чтобы зафиксировать эту разницу, граф строится в нотации split-node: каждый элемент курса расщепляется на пару узлов «доступ» и «завершение». Между ними проводится внутреннее ребро (нельзя завершить, не получив доступ), а зависимости от политик становятся типизированными рёбрами между узлами разных элементов. Трансформация показана на рисунке 2.11 на примере цикла из двух взаимных требований завершения.

На split-node графе цикл обнаруживается стандартной процедурой выделения сильно связных компонент. Любая сильно связная компонента размером больше одного узла соответствует циклу, проходящему через её узлы. Алгоритм возвращает методисту список таких циклов с конкретными политиками, замыкающими цепочку.

Сложность построения графа линейна по числу политик, поскольку каждая политика добавляет фиксированное число рёбер; обнаружение сильно связных компонент имеет линейную сложность по числу узлов и рёбер графа.

### 2.4.2 Конвейер логического вывода

Конвейер логического вывода обслуживает запросы двух видов: проверку доступа и верификацию курса. Устроен как четырёхстадийная последовательность (рисунок 2.12).

Стадия предобработки обновляет вспомогательные индивиды онтологии: слой Python-логики пересоздаёт `current_time_ind` и семейство индивидов `AggregateFact` в соответствии с правилами раздела 2.3.1. Затем резонер прогоняет SWRL-правила и DL-аксиомы над онтологией, выводя свойства `satisfies` (условие выполнено) и `is_available_for` (элемент доступен). После вывода применяется гипотеза закрытого мира и каскадная блокировка по иерархии (раздел 2.3.2). Завершающая стадия материализации записывает результаты в кэш.

Запуск резонера ограничен таймаутом; превышение переводит ответ в статус «частично рассчитано» с использованием предыдущего материализованного значения. При недоступности кэша конвейер не отказывает: рассчитанные значения возвращаются напрямую, минуя стадию материализации.

### 2.4.3 Алгоритм восходящей агрегации завершённости

Восходящая агрегация завершённости преобразует факты завершения отдельных активностей в факты завершения родительских контейнеров (модулей и курса). Без этого механизма обучающийся, завершивший все обязательные активности модуля, формально не считается завершившим модуль; поэтому ломаются зависимости вида «доступ к модулю Y разрешён, если завершён модуль X».

Алгоритм обходит дерево курса снизу вверх. Для каждого модуля проверяется, завершены ли все его обязательные потомки; если да, в онтологию добавляется факт завершения модуля. После обработки всех модулей курса аналогично проверяется завершённость самого курса. На рисунке 2.13 показан пример восходящего каскада на демонстрационном курсе из раздела 2.2.3: завершение всех обязательных активностей двух модулей даёт завершение этих модулей, а наличие ещё одного незавершённого модуля сохраняет курс в незавершённом состоянии.

Алгоритм идемпотентен: повторный запуск на тех же фактах не меняет результат. Сложность линейна по числу узлов в дереве курса. Инвариант корректности: контейнер считается завершённым тогда и только тогда, когда все его обязательные потомки завершены, и это свойство сохраняется при добавлении или удалении атрибута обязательности у любого потомка с пересчётом.

### 2.4.4 Алгоритм анализа достижимости элементов

Алгоритм анализа достижимости элементов выявляет элементы курса, к которым ни один обучающийся не может получить доступ ни при каком прогрессе и ни при какой принадлежности к группе. Такая ситуация возникает при ошибке методиста при настройке политик: например, политика требует оценки выше 100 баллов, или временное окно с пустым пересечением, или цепочка зависимостей замкнута без точки входа. Алгоритм обнаруживает все такие случаи на этапе верификации до того, как обучающийся столкнётся с заблокированным элементом.

Анализ устроен как последовательность из трёх проходов с разной семантикой. Краткое содержание проходов приведено в таблице 2.4.

Таблица 2.4 – Три прохода анализа достижимости

| Проход                                       | Что проверяет                                                                                                                              | Инструмент                                                       | Сложность                                            |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------- | ------------------------------------------------------------- |
| 1. Атомарная выполнимость     | Каждое атомарное условие в отдельности (порог, временное окно)                                     | Проверка значений атрибутов политики      | Линейная по числу политик               |
| 2. Структурная достижимость | Существование пути в графе зависимостей до элемента от любого корня                          | Графовый обход с фиксированной точкой     | Линейная по числу узлов и рёбер     |
| 3. Совместная выполнимость   | Существование набора фактов, при котором все условия пути одновременно выполнимы | DL-резонер на синтетических утверждениях | Экспоненциальная в худшем случае |

Первые два прохода составляют основную процедуру; они обнаруживают подавляющее большинство практических случаев недостижимости с гарантированным временем работы. Третий проход применяется только к элементам, не помеченным первыми двумя и защищённым сложной комбинацией политик; он перенесён в перспективу развития работы из-за высокой стоимости и редкости подобных случаев.

На полном множестве элементов курса алгоритм возвращает методисту список недостижимых элементов с указанием прохода, обнаружившего недостижимость, и причины. Связь со свойством ацикличности такая: цикл в графе зависимостей делает все элементы цикла недостижимыми, поэтому А4 находит то, что мог бы пропустить А1; обратное также верно: атомарно невыполнимая политика даёт недостижимость без цикла, что А1 не обнаруживает.

## 2.6 Модели данных и интерфейсы интеграции

### 2.6.1 Спецификация программного интерфейса

Программный интерфейс системы спроектирован поверх протокола HTTP и описан в спецификации OpenAPI. Внешние клиенты (клиентское приложение методиста и интегрирующиеся СДО) обращаются к шести группам эндпоинтов, по одной на функциональную линию: управление политиками, проверка доступа, события прогресса, верификация курса, симулятор, интеграция с СДО. Каждой группе соответствует один контроллер из компонентной декомпозиции (раздел 2.1.3).

Все запросы и ответы используют структуры данных в формате JSON, описанные через Pydantic. Для каждой структуры зафиксированы обязательные и опциональные поля, типы, ограничения значений; при нарушении контракта запрос отклоняется со структурированной ошибкой, содержащей путь к нарушенному полю и причину отклонения. Контракт ошибок единый для всех эндпоинтов и включает код ошибки, краткое описание, технические детали для отладки и идентификатор запроса для трассировки.

### 2.6.2 Схема кэша и стратегия инвалидации

Кэш разработанной системы организован как хранилище пар «ключ-значение» с поддержкой времени жизни записей. Ключи разделены на несколько пространств имён по типам кэшируемых данных: решения о доступе, отчёты верификации курса, результаты восходящей агрегации, граф зависимостей. Полезная нагрузка хранится в формате JSON со структурированными полями, что упрощает частичные обновления и чтение при отладке. Ёмкость кэша рассчитана на типовую нагрузку рабочей системы из нескольких тысяч активных обучающихся; при превышении ёмкости применяется политика вытеснения по принципу наименьшей последней использованности.

Кэш материализованных решений о доступе ускоряет ответ на типичный запрос с миллисекунд логического вывода до десятка микросекунд чтения.

При каждом событии, изменяющем онтологию, инвалидируется заранее определённый набор ключей. Сводка приведена в таблице 2.5.

Таблица 2.5 – Стратегия инвалидации кэша

| Событие                                                                                  | Инвалидируемые ключи                                                                                                                   |
| ----------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Создание, изменение или удаление политики                   | Все решения для целевых элементов политики и кэшированный отчёт верификации курса  |
| Событие прогресса обучающегося (завершение, оценка) | Все решения для этого обучающегося и кэшированные результаты восходящей агрегации |
| Импорт курса или изменение его структуры                     | Все решения и отчёты по курсу, а также кэшированный граф зависимостей                           |
| Изменение глобальных параметров системы                     | Полный сброс кэша                                                                                                                          |

Время жизни записей кэша задано фиксированным; при истечении времени жизни запись удаляется автоматически без дополнительной координации с источником истины. Дополнительно при политиках типа «временное окно» дат-границы выравниваются на целые часы; это гарантирует, что к моменту истечения времени жизни записи кэша все актуальные временные переходы политик уже произошли. Решение «фиксированное время жизни плюс выравнивание границ» взято вместо более сложной адаптивной стратегии расчёта ближайшей временной границы.

Второй слой задаёт соответствие между параметрами политики разработанной системы и параметрами availability conditions целевой платформы для каждого из девяти типов правил. На третьем слое отображены сущности данных: пользователи, группы, прогресс, оценки и компетенции находят соответствие в таблицах целевой платформы.
