# SAT_CODE_GUIDELINES — стандарты кода

> **Назначение.** Свод правил, применяемый при любой правке кода. Загружается каждой сессией Claude Code, когда задача затрагивает `code/`. Ссылается на PROJECT_BIBLE §3.6 (диф), SAT_DATA_MODELS (TBox/SWRL), SAT_ALGORITHMS (А1–А6).
>
> **Дата создания:** 21.04.2026. **Последняя правка:** 21.04.2026.

## 1. Архитектурная дисциплина

Жёсткие запреты. Нарушение — блокирующий комментарий в ревью.

- **Не писать бизнес-логику в роутерах.** Роутер — маппинг HTTP ↔ сервис, обработка ошибок, коды ответа. Любые условия, циклы по онтологии, вызовы reasoner-а — в сервисе.
- **Не обращаться к Owlready2 напрямую из API Layer.** Только через `OntologyCore` или сервисы. Исключение — `_get_node_label` и подобные helper-ы на `OntologyCore`, вызываемые контроллерами для форматирования.
- **Не запускать `sync_reasoner_pellet` вне `ReasoningOrchestrator`.** Любой reasoning проходит через `core.run_reasoner()` → orchestrator. Прямой вызов ломает pre-enrich (LIM1, LIM2) и CWA.
- **Не удалять `is_available_for` / `satisfies` выборочно.** Только полная очистка перед прогоном reasoner-а (`clear_inferred_triples`). Точечная правка монотонной OWL-ной онтологии даёт «фантомные» выводы (LIM6).
- **Не добавлять SWRL-правила, выводящие `is_available_for` напрямую.** Стандарт — двухуровневая семантика: атомарный шаблон → `satisfies`, мета-правило → `is_available_for`. См. SAT_DATA_MODELS §2.1.
- **Не передавать OWL-индивиды в API/schemas.** Сервис маппит в dict/Pydantic до возврата. Индивиды не сериализуются корректно и тянут lifecycle онтологии.
- **Не хранить состояние между запросами в сервисах.** Только в `OntologyCore` (onto, redis, репозитории). Сервисы — stateless.
- **Не создавать обход слоёв.** API → Service → Core → Repository/Onto. Service → Repository напрямую — OK. Repository → Service — нет.

## 2. Reasoning-дисциплина

- **Всегда сохранять онтологию ПОСЛЕ успешного reasoning.** При `result.status != "ok"` — не сохранять, откатывать ABox (НФТ-5).
- **Перехватывать `ReasoningResult.status`**, а не `try/except Exception` вокруг `run_reasoner()`. Статусы: `ok` / `inconsistent` / `timeout` / `error`. Каждый — отдельная HTTP-реакция.
- **Не глотать исключения Pellet молча.** Лог уровня `warning` + re-raise или структурированный возврат. Глухие `except: pass` запрещены.
- **`check_consistency` — предикат, не метод-побочный эффект.** Вызывающий код сам решает, откатывать ли изменение.
- **Таймаут reasoning** настраивается через `ReasoningOrchestrator(timeout_sec=...)`, default 10 с (НФТ-3). Не хардкодить в сервисах.
- **CWA-enforcement — post-processing, не онтология.** NAF в SWRL писать нельзя (LIM5). Default-deny — в `AccessService` после чтения `is_available_for`.

## 3. Owlready2 и OWL-специфика

- **FunctionalProperty mixin ⇒ scalar API.** `indiv.prop = value`, `indiv.prop` возвращает скаляр. Не `[value]`, не `prop[0]`.
- **Non-functional property ⇒ list API.** `indiv.prop = [v1, v2]`, `indiv.prop[0]`. Даже для одного значения — список.
- **Перед `destroy_entity` убеждаться, что индивид не ссылается из других мест** — иначе висячие ссылки. Порядок: снять inverse-связи, потом destroy.
- **Reified factы (CurrentTime, AggregateFact)** создаются и уничтожаются только в `ontology_enricher`. Никакой другой модуль их не трогает.
- **Имена индивидов — `snake_case`**, без дефисов и точек. Owlready + Pellet чувствительны к IRI.
- **`search_one(iri=f"*{id}")` без `type=...` запрещено.** В ABox ID коллидируют между классами (methodist_1 и comp_python могут иметь одинаковый суффикс). Всегда указывать `type=onto.Competency` (или нужный класс).

## 4. SWRL

- **Новое правило выводит только `satisfies(?s, ?p)`.** Единственное исключение — мета-правило ступени 2 и вспомогательные H-* (наследование компетенций).
- **Любой атом с внешним/производным значением → enricher.** Даты, агрегаты, счётчики попыток — реификация через `CurrentTime` / `AggregateFact` / …, не попытка впихнуть SWRL-built-ins с side-effect.
- **Каждый новый шаблон сопровождается тестом** в `tests/onto/test_swrl_*.py`: минимальный ABox, ожидаемый `satisfies`, негативный случай.
- **DL-safety сохраняется.** Все переменные — named individuals. Никаких existential-переменных в head.
- **`DifferentFrom` обязателен в композитных правилах** с двумя и более `has_subpolicy` — иначе SWRL унифицирует переменные и AND вырождается в OR.

## 5. Комментарии и документация в коде

### Общие правила

- **По умолчанию комментарий не пишем.** Имя функции/переменной обязано объяснять что.
- **Комментарий пишем только когда непонятно ПОЧЕМУ.** Неочевидный инвариант, ограничение инструмента/платформы, workaround, причина необычного решения.
- **Никаких ссылок на внутренние документы проекта в коде.** `PROJECT_BIBLE`, `SAT_*`, `LIM1`, `НФТ-5`, `FIX3`, «решение 21.04», `§3.6` — это артефакты нашей работы, в коде их быть не должно. Если надо сослаться на ограничение — назови его по сути: «OWL monotonic, no TMS» вместо «LIM6»; «open world → no NAF» вместо «LIM5»; «cache miss target 2s» вместо «НФТ-2». Код должен читаться как самостоятельный артефакт, без кросс-ссылок в нашу документацию.
- **Русский язык, безличная форма.** «Очистка выводов перед прогоном», не «Очищаем». «Проверка на цикл», не «Проверяем».
- **Человекоподобный тон, без LLM-паттернов.** Не писать «🚀», «✨», «Давайте...», «Важно отметить, что...», «Для этого мы...», «Шаг 1: ... Шаг 2: ...», «=== СЕКЦИЯ ===» в качестве разделителей. Комментарий должен выглядеть так, будто его написал уставший инженер в 18:00, а не модель, которая хочет понравиться.
- **Короткий комментарий — на той же строке, что и код.** Если помещается — inline. Над строкой — только если комментарий не умещается в пределах разумной ширины (~120 символов).
- **Одна строка на комментарий.** Блочный комментарий — красный флаг: или имя переменной плохое, или код слишком сложный и надо рефакторить.
- **Количество строк в файле = количество строк кода.** Если в файле 200 строк и из них 80 — комментарии, что-то не так. Комментарии — исключение, не фон.
- **Не дублировать сигнатуру в docstring.** `def delete_policy(policy_id: str) -> bool` не нуждается в docstring «удаляет политику по ID». Docstring пиши, только если есть нетривиальный инвариант, thread-safety, побочные эффекты или неочевидный протокол использования.
- **`TODO` — только с именем и сутью.** Формат: `TODO(bulat): joint-satisfiability проход`. Без дат (они тут же устаревают) и без ссылок на внутренние §.
- **Никаких «заглушек» в комментариях.** Код либо работает, либо выкидывает `NotImplementedError` с ясным сообщением. «Здесь в будущем можно добавить...» — удалять.

### Что не писать

```python
# создаём политику                            ← очевидно из имени
policy = onto.AccessPolicy(policy_id)

# === 1. Инициализация ===                    ← визуальный шум
# === 2. Обработка ===

# мы теперь проверяем совместимость            ← первое лицо, LLM-паттерн
if not consistent: ...

# Здесь в будущем можно добавить: ...         ← аспирационная пустота

# LIM1 (SWRL не имеет now()) — инжекция        ← внутренний ярлык в коде
enrich_current_time(onto)

# FIX3 / §3.6.7 — чиним поиск                 ← ссылка в нашу документацию
parent = onto.search_one(...)

# ✨ Запускаем reasoning!                      ← эмодзи, LLM-тон
run_reasoner()
```

### Что писать

```python
enrich_current_time(onto)  # SWRL не имеет now(), подкладываем текущее время

fact.for_student = student  # functional property — скалярный API

if result.status == "inconsistent":
    self._rollback_policy(new_policy, source)  # ABox частично изменён, откатываем
    raise PolicyConflictError(result.error)

# OWL монотонен и не поддерживает truth maintenance, поэтому перед
# каждым прогоном очищаем выводы целиком, а не точечно
clear_inferred_triples(onto)
```

### Стиль docstring-ов

- Первая строка — одно предложение в безличной форме, точка в конце.
- Дальше — только если необходимо: описание инварианта, ссылка на Bible/SAT, пример нетривиального использования.
- Args/Returns/Raises пишем, только если тип не покрывает смысл параметра. Простые `student_id: str` без описания — норма, избыточно.

Пример допустимого docstring:

```python
def reason(self) -> ReasoningResult:
    """Pre-enrich, запуск Pellet, подсчёт выводов.

    При status=inconsistent ABox уже изменён — откат на совести вызывающего.
    """
```

Внутренние ссылки (`§2.4`, `SAT_ALGORITHMS`, `LIM*`) в docstring тоже не пишем.

## 6. Тесты

### Целевое покрытие

- **≥ 60%** для всего backend (НФТ-11, критерий К2).
- **≥ 70%** для Core Layer: `reasoning_orchestrator`, `ontology_enricher`, `graph_validator`, `verification_service` (когда появится). Эти модули — научный вклад, требования выше базовых.
- Фронт — smoke + snapshot UC-1/4/6/7. Unit-покрытие не требуется.

### Разделение

```
code/backend/src/tests/
├── unit/           — без Pellet, без Redis, моками индивидов
├── integration/    — с реальным Pellet, с in-memory онтологией
└── fixtures/       — shared ABox (demo + негативные мини-ABox из SAT_DATA_MODELS §6)
```

Unit-тесты должны пробегать за < 2 с на весь набор. Integration — до 30 с (Pellet — ~1 с на прогон).

### Правила написания

- **Именование:** `test_<что>_<условие>_<ожидание>`. Пример: `test_create_policy_cyclic_raises_policy_conflict_error`.
- **Arrange-Act-Assert** — три блока, разделённых пустой строкой. Комментарии `# arrange` и т.д. не нужны, достаточно структуры.
- **Один тест — одна проверка.** Не валить 5 ассертов в один `test_happy_path`.
- **Фикстуры — только для shared state.** Одноразовые данные — в теле теста.
- **Не моковать Pellet.** Он — часть системы, тестируется as-is. Мокать — только внешние зависимости (Redis, HTTP).
- **Негативные ABox из SAT_DATA_MODELS §6.8** — использовать как fixture для тестов верификации (EXP1/EXP2).

### Процесс

- Новый функционал: тест пишем параллельно с кодом. Если тест не написан — задача не COMPLETE.
- Поправка бага: сначала failing-тест, потом фикс. «Regression by coverage».
- Существующий падающий тест не игнорировать через `@pytest.mark.skip`. Либо чинить, либо удалять с записью в Bible §7 («тупик»).
- После каждой задачи: `pytest --cov` локально. Падения — до перехода к следующей задаче.

## 7. Обработка ошибок

- **Catch specific, not `Exception`.** Пример: `except PolicyConflictError`, `except ReasoningTimeoutError`. Широкий `except Exception` допустим только в роутере как last-resort с HTTP 500.
- **Логируем ДО re-raise.** `logger.exception(...)` + `raise` — нормально. Просто `raise` без лога — теряем trace на проде.
- **HTTPException — только в роутере. Строго.** Сервис кидает доменное исключение (`PolicyConflictError`, `PolicyNotFoundError`, ...), роутер ловит и маппит в HTTP-код. Сервис не знает про HTTP.
- **Откат при частичных изменениях.** Если операция включает несколько write-шагов (create_policy → reason), и поздний шаг падает — откатывать ранние, чтобы состояние системы осталось согласованным.

## 8. Python-стиль

- **Type hints на публичном API** обязательны (методы класса, module-level функции). Приватные `_method` — по желанию.
- **`dataclass` для value-объектов** (результаты, конфиги). Никаких голых dict-ов в сигнатурах, кроме сериализации на границе API.
- **`Enum` для фиксированного набора значений** (rule_type, Status). Магические строки `"completion_required"` в коде — только через `RuleType.COMPLETION.value`.
- **F-strings, не `.format` или `%`.**
- **Короткие функции.** Если метод > 40 строк — скорее всего ему нужен рефакторинг.
- **`from __future__ import annotations`** — используем по необходимости (forward refs, циклические импорты, `dict[str, int]` в Py 3.9). Без фанатизма.

## 9. Дисциплина коммитов (из корневого CLAUDE.md)

- **Никаких auto-commit.** Коммит делает пользователь по подтверждению.
- **Никаких `--no-verify`** (pre-commit hooks — часть контроля качества).
- **Один логический коммит — один FIX или один блок §3.6.7.** Не валить Блок 1 целиком в один мегакоммит.
- **Сообщение коммита ссылается на Bible.** Пример: `Блок 1.1–1.2: TBox T1–T8/T11/T13 + SWRL S1–S4 (satisfies + мета-правило)`.

## 10. Performance и масштабирование

- **НФТ-1** (cache hit, ≤50 мс): все операции чтения доступа — через `CacheManager` сначала.
- **НФТ-2** (cache miss, ≤2000 мс): reasoning полного курса (500 элементов, 1000 студентов, 100 правил) должен укладываться.
- **НФТ-4 sizing.** Профилирование через `pytest-benchmark` в integration-тестах. Если reasoning >3 с на стандартном курсе — остановиться и разобраться, не продолжать.
- **Redis — оптимизация, не источник истины.** При `redis=None` система работает (default-deny + каждый раз reasoning). Тесты должны проходить без Redis.

## 11. Definition of Done для задачи фазы 2

Задача НЕ считается `COMPLETE`, пока:

- [ ] Код написан и проходит статическую проверку (`python -c "import <module>"`).
- [ ] Тесты написаны и проходят (`pytest path/to/tests/ -q`).
- [ ] Покрытие по модулю соответствует п. 6 (≥60% в целом, ≥80% для Core Layer).
- [ ] Изменён соответствующий раздел Bible (§3.6 статус пункта, §6 чекбокс фазы, §7 если решение).
- [ ] Если добавлен новый публичный класс/метод — появилась строка в SAT_DATA_MODELS/SAT_ALGORITHMS или ссылка на существующую.
- [ ] Нет новых `TODO` без имени и даты.
- [ ] Нет закомментированного кода. Удалённое — удалено, не спрятано в `#`.
